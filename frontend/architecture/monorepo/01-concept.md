# 1. 모노레포

---

## 1.1 개념

여러 개의 프로젝트를 하나의 버전 관리 저장소에서 관리하는 것을 의미한다.

## 1.2 쓰는 이유

기존에는 프로젝트마다 저장소를 분리해서 관리 했다.

이 방식에서는 공통 코드를 수정할 때마다 여러 저장소를 오고가야 하고, 패키지를 배포하고, 각 프로젝트에서의 버전을 맞춰야 했다.

프로젝트 수와 규모가 커질수록 이 과정에서 실수가 많이 일어나면서 모든 프로젝트를 한 저장소에서 관리하는 방식인 모노레포가 등장했다.

## 1.3 장점

1. 공통된 코드를 관리가 쉽다.
2. 프로젝트 간의 버전 관리가 쉬워진다.

## 1.4 단점

1. 의존성 관리, 빌드 순서, 폴더 구조를 처음부터 설계해야 하기 때문에 초기 설정이 복잡하다.
2. 레포가 커질수록 관리가 어려워 진다.

## 1.5 폴더 구조 예시

```jsx
monorepo/
├── apps/                # 독립적으로 배포 및 실행되는 애플리케이션
│   ├── web/             # Next.js 또는 React 웹 서비스
│   ├── admin/           # 관리자 사이트
│   └── docs/            # 문서 사이트
├── packages/            # 여러 앱에서 공유하는 라이브러리 및 설정
│   ├── ui/              # 공통 UI 컴포넌트 (디자인 시스템)
│   ├── utils/           # 유틸리티 함수
│   ├── eslint-config/   # 공유 린트 설정
│   └── tsconfig/        # 공유 TypeScript 설정
├── package.json         # 루트 프로젝트 설정 및 도구 관리
├── pnpm-workspace.yaml  # pnpm 사용 시 워크스페이스 정의
└── turbo.json           # Turborepo 빌드 캐시 및 파이프라인 설정
```
