# 4. Nx

---

## 4.1 개념

모노레포 환경에서 여러 개의 애플리케이션과 패키지 사이의 관계를 지능적으로 파악하여 프로젝트 구조 시각화, 개발, 빌드, 테스트 속도를 극대화하는 관리 도구이다.

## 4.2 나오게 된 이유

1. 무식한 전체 빌드 해결: 이전에는 코드 한 줄만 고쳐도 모든 프로젝트를 다 빌드해야 했습니다. 이로 인한 시간 낭비를 막고 바뀐 부분만 빌드하기 위해 등장했다.
2. 영향 범위 파악 자동화: 이전에는 수정 시 어디가 고장 날지 사람이 직접 찾아야 했다. 이를 의존성 그래프로 자동 분석하여 사고를 막기 위해 탄생했다.
3. 중구난방 설정 통일: 프로젝트마다 제각각이었던 빌드, 테스트 설정을 코드 생성기로 표준화하여 관리 효율을 높이려 했다.
4. 중복 작업 제거: 동료가 이미 한 빌드를 내 컴퓨터에서 또 하는 낭비를 없애고, 캐싱하기 위해 만들어졌다.

## 4.3 장점

1. 스마트 캐싱: 이전에 수행한 빌드나 테스트 결과를 저장해둔다. 코드가 변하지 않았다면 다시 작업하지 않고 저장된 결과물을 즉시 가져온다.
2. Affected 명령: 수정된 코드와 연관된 프로젝트만 골라내어 테스트하거나 빌드한다. 관련 없는 프로젝트는 건드리지 않아 자원을 아낀다.
3. 그래프 시각화: `nx graph` 명령어로 전체 프로젝트가 어떻게 얽혀 있는지 시각적인 지도로 보여준다.
4. 코드 생성 자동화: 팀의 표준에 맞는 앱이나 라이브러리 구조를 명령어 하나로 똑같이 생성하여 코드 품질을 통일합니다.

## 4.4 단점

1. 높은 학습 곡선: Turborepo나 단순 워크스페이스에 비해 개념이 방대하여 익히는 데 시간이 걸린다.
2. 복잡한 초기 설정: 프로젝트 규모가 작을 때는 설정 파일들이 너무 많고 복잡하게 느껴질 수 있다.
3. Nx 전용 생태계 의존: Nx가 제공하는 플러그인 방식을 깊게 사용할 경우, 나중에 다른 도구로 옮기기가 매우 까다로울 수 있다.
