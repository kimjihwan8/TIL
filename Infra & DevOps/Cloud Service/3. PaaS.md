# 개념
IaaS는 컴퓨터를 빌린 다음에 내가 쓸 것들을 하나하나 정해줘야 하는 반면, PaaS(Platform as a Service)는 이름에서 부터 알수 있듯이 이미 개발에 필요한 모든 인프라 세팅이 끝난 플랫폼을 빌려 쓰는 느낌이다.
# 특징
- 가상 컴퓨터 빌리기, 운영체제 설치, 자바/파이썬/Node.js 환경 세팅, 네트워크 및 방화벽 설정 등이 공급업체에 의해 이미 다 되어 있다.
- 따라서 개발자는 인프라 환경에는 신경을 쓸필요 없이, 내가 짠 코드를 올리기만 하면 된다.
# 장점
1. 딸깍 배포:
	- 리눅스 명령어 입력하고, Nginx 세팅하고, 도커 이미지 빌드할 필요가 전혀 없다. 내가 만든 프론트엔드 코드를 Github에 올리기만 하면, PaaS(Vercel 등)가 알아서 빌드하고 배포해준다.
		- 추가로 프론트엔드 특화 PaaS 들은 기본적으로 cdn까지 내장되어있어서 배포를 되면 알아서 글로벌 엣지 서버로 코드가 퍼진다.
2. 유지보수 해방
	- OS 보안 업데이트, SSL 인증서 발급 및 갱신 등을 클라우드 회사가 알아서 다 해준다.
3. 모니터링: 
	- 방문자가 얼마나 오는지, 서버에 에러가 없는지를 별다른 세팅없이 대시보드로 다 보여준다.
# 단점
1. 자유롭지 않음
	- 운영체제 버전을 바꾸고 싶다거나, 데이터베이스 설정 바꾸고 싶을때가 있을수도 있는데 이건 불가능하다. 플랫폼이 정해준거 안에서만 놀아야 한다.
2. 트래픽이 많아지면 비싸짐
	- 서비스 규모가 커져서 트래픽이 많아지면, 똑같은 서버 스펙 대비 IaaS(AWS EC2 등)보다 비용이 수배 이상 훨씬 비싸진다. 편리함을 제공하는 대신 마진을 엄청 남기기 때문이다.
3. 종속
	- Vercel이나 Heroku 같은 특정 PaaS 시스템에 맞춰 코드를 최적화해 놓으면, 나중에 클라우드로를 다른걸로 변경할때 코드를 뜯어고쳐야 하거나 아예 변경이 불가능해지는 종속 현상이 생긴다.
