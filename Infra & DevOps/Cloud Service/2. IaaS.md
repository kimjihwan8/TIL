# 개념
아예 아무것도 없는 빈 공간만 대여받는 느낌이다. 즉, 가상의 컴퓨터만 빌리는 것이다.
대표적인 예시로 AWS EC2가 있다.
# 특징
- 대여한 컴퓨터에 내가 원하는 운영체제를 직접 고르고, 그 위에 필요한 프로그램을 일일히 수동으로 깔아야 한다. 자유도가 매우 높은 만큼 세팅할 게 많다.
# 장점
1. 컴퓨터 세팅을 내 마음대로 할수 있음
	- 인프라 하드웨어만 빌려주기 때문에, 그 위에 어떤 운영체제를 깔든, 어떤 데이터베이스와 런타임 등을 쓰든 내 마음대로 완벽하게 커스텀할 수 있다.
	- 회사 고유의 엄격한 보안 정책이나 특수한 네트워크 설정을 그대로 적용할 수 있다.
2. 돈 아끼기
	- 온프레미스처럼 컴퓨터를 미리 사둘 필요 없이, 오직 컴퓨터를 켜놓은 시간만큼만 돈을 내니까 엄청 합리적이다.
3. 트래픽 몰려도 안 터짐
    - 트래픽이 몰리면 오토 스케일링을 통해 서버 자원을 유동적으로 늘려 서버 펑을 방지할 수 있다.
4. 물리적으로 관리할 필요없음
	- 실제 서버 컴퓨터가 터지거나 고장 나면 AWS 같은 대기업에서 알아서 새 걸로 고치고 바꿔준다.
# 단점
1. 인프라 모르면 시작도 못 함
	- 컴퓨터만 덜렁 던져주기 때문에 리눅스 명령어, 포트 설정, 방화벽, 웹 서버 세팅 같은 인프라 지식을 모르면 시작도 못한다. 진입장벽이 꽤 높다.
2. 유지보수
	- OS 보안 업데이트, 데이터베이스 백업, SSL 인증서 발급 및 갱신 등을 전부 내가 주기적으로 수동 관리해 줘야 한다. <- 이 귀찮음 때문에 결국 PaaS나 다른것들을 찾게 됨.
3. 해킹
	- 보안 설정을 대충 해두면 해커들이 들어와서 내 서버를 몰래 비트코인 채굴기로 써먹거나 전체 시스템을 날려버릴 위험이 있다.

# Docker를 쓰는 이유
내가 원하는 운영체제와 프로그램들을 깔아서 세팅해 놨는데, 이 컴퓨터 저 컴퓨터 옮겨 다닐 때마다 그 짓을 똑같이 반복해야 한다는 점이다. 심지어 내 맥북에서는 잘만 돌아가던 코드가 막상 IaaS 서버로 가면 환경이 미묘하게 달라서 에러가 터진다.
- 매번 서버 컴퓨터를 새로 빌릴 때마다 필요한 프로그램과 환경 설정을 처음부터 일일이 까는 것은 매우 귀찮고 번거롭다.
    
- 이 문제를 해결하기 위해, 내가 필요한 개발 환경과 프로그램을 '도커 컨테이너(Docker Container)'라는 하나의 상자에 한 번에 예쁘게 포장해서 IaaS 컴퓨터에 툭 던져 실행하는 방식을 사용한다.
