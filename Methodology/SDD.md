SDD는 실제 비즈니스 로직이나 UI를 코딩하기 전에, 제품의 스펙(Specification, 명세)을 먼저 정의하고 이를 강제하는 환경을 구축한 뒤 개발을 진행하는 방법론이다.

과거에는 백엔드, 디자이너와 소통하기 위한 지도였다면, AI 에이전트가 코드를 짜주는 지금은 AI가 내 의도를 벗어나지 않고 고품질 프론트엔드 코드를 짜도록 통제하는 인터페이스로 진화했다.
## 1. 왜 프론트엔드에 SDD가 필요한가?
- 백엔드: 백엔드 API가 완성될 때까지 프론트엔드는 손을 놓고 기다리거나, 나중에 API 필드명이 바뀌면 프론트 코드를 다 뜯어고쳐야 했다.
- 디자인: 기획서, 피그마와 미묘하게 다르게 구현된 UI 때문에 기획자, 디자이너와 무한 수정을 반복했다.
- 바이브 코딩: claude code에게 장바구니 기능 만들어줘라고 대충 채팅으로 지시하면, 프로젝트의 공통 컨벤션을 깨뜨리거나 예외 처리가 누락된 레거시 코드를 양산한다.

## 2. 프론트엔드 SDD의 3대 핵심 실천 방식
현대 프론트엔드 개발자가 SDD를 실천하는 방법은 크게 3가지 계층으로 나뉜다.
### 2.1 데이터 계층: API 스펙 주도
백엔드와 말로만 맞추는 게 아니라, `OpenAPI(Swagger) 스펙 문서`를 먼저 발행한다.

- 이 문서를 기반으로 프론트엔드는 TypeScript 인터페이스/타입을 자동으로 생성(Generate)한다.
- API가 없어도 가짜 데이터를 뱉어주는 Mock 서버(MSW)를 띄워 프론트엔드 개발을 백엔드보다 먼저 완벽하게 끝낸다.
- 효과: 타입이 스펙을 강제하므로 API 연동 에러가 전면 차단된다.

### 2.2 UI 계층: 디자인 시스템 스펙 주도
화면 전체를 무작정 그리는 게 아니라, 디자이너와 합의한 디자인 시스템 스펙(피그마 등)을 기반으로 독립적인 컴포넌트 명세를 작성한다.

- 브라우저에 화면을 띄우기 전에, `Storybook`을 켜고 컴포넌트(Button, Modal 등)가 가질 수 있는 모든 상태(Props)를 명세서처럼 등록한다.
- 효과: 컴포넌트 자체가 하나의 완벽한 '시각적 스펙 문서'가 되어 기획자/디자이너와 화면 없이도 컴포넌트 단위 검증이 가능하다.

### 2.3 구현 계층: AI 에이전트 스펙 주도 (with Spec.md)

AI에게 무작정 코딩을 시키는 게 아니라, 구현할 화면의 규칙을 마크다운 문서로 명확히 정의하고 AI에게 넘긴다.

- `todo-spec.md` 파일에 "FSD 아키텍처를 따를 것", "컴포넌트는 Tailwind CSS로 짤 것", "인풋 창에 글자가 없을 땐 등록 버튼을 disabled 처리할 것" 같은 명세를 세밀하게 적는다.
- AI 에이전트(Cursor, Claude Code)는 채팅창이 아닌 이 스펙 문서를 최상위 지침(Single Source of Truth)으로 삼아 소스 코드와 테스트 코드를 자동으로 짜낸다.
- 효과: 맥락이 흐려져 AI가 헛소리(Hallucination)를 하는 것을 막고, 인간은 코더가 아닌 '스펙 검수자'로 포지셔닝한다.
